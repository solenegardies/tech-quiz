# Server rules

- **Repository pattern**: Routes never call Prisma directly — all DB access goes through repository classes in `src/repositories/`.
- **Tests are mandatory**: Every module must have tests. Only modify existing tests if behavior genuinely changed — if tests break, treat it as a bug in your code.
- **No `return await`** unless inside a `try/catch`.
- **External enums / mapper drift**: When code depends on external enum-like values, categories, statuses, or mapper inputs from third-party systems, handle known values explicitly and `logger.error` on unseen values.

## Testing rules (server)

Tests aim to be **near-integration**: real internal logic runs, only external boundaries are mocked.

**Always mock (external boundaries):**

- `lib/prisma` → mock `PrismaClient`
- `lib/logger` → mock Winston logger
- `argon2` (CPU-intensive crypto lib)
- Email sending functions (`lib/email/emailService.ts`)

**Never mock (internal logic must run for real):**

- Session functions: `createSession`, `setSessionCookie`, `validateSessionToken`
- Token functions: `consumePasswordResetToken`, `issuePasswordResetToken`, `hashSecret`
- Repositories (`UserRepository`, `EmailVerificationTokenRepository`, etc.)
- Utility functions: `getClientIp`, `getUserAgent`, `normalizeEmail`, etc.
